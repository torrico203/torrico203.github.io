1. 글로벌 결제 및 수익화 인프라 구축

Xsolla PayStation v3 직접 연동 functions/xsolla.js (627 LOC) · managers/xsollaManager.js (424 LOC)

  • 결제 토큰 발급 / Webhook / 재화 소비의 3단 파이프라인을 Firebase Cloud Functions (asia-southeast1, concurrency 80) 로 직접 구현. 4개 엔드포인트: createXsollaToken, xsollaWebhook, consumeGemForContent, consumePointForContent.
  • PG v2→v3 마이그레이션 트러블슈팅 실적: 한국 결제수단에서 발생한 Xsolla v2 Error 2002 (item not found) 이슈를 purchase.virtual_items SKU 처리 방식의 v3 차이를 분석해 purchase.items[].sku/quantity 기반 + sandbox top-level 플래그 + Basic Auth (project_id:api_key) 구조로 전환하여 해결 (xsolla.js:98-111 주석에 트러블슈팅 로그 보존).
  • Fraud Prevention / 무결성 설계 4종:
    1. 클라이언트 가격 불신 원칙 — 가격·수량은 Firestore gem_packages/{id} 단일 출처(SSOT)에서 강제 조회 (xsolla.js:81-87).
    2. Sandbox 모드 서버 단일 진실 공급원 — 클라이언트가 sandbox 플래그를 송신할 수 없도록 차단 (xsolla.js:24-25).
    3. SHA-1 Webhook 서명 검증 — XSOLLA_PROJECT_KEY Firebase Secret 기반 서명 매칭 후에만 재화 지급 (xsolla.js:210-227).
    4. 밴 계정·중지 상품 사전 차단 + externalId = uid_${timestamp} 멱등성 키로 중복 지급 방지.
  • 국가/통화 처리: user.country (KR 기본) + billing.purchase.total.currency (KRW 기본) — Xsolla 가 200+ 국가 결제수단을 핸들링하는 구조에 맞춰 user.id/email/country 페이로드를 표준화.

AdSense × Xsolla 듀얼 수익화 전략

  • AdSense 는 managers/adManager.js (440 LOC) 에서 8개 슬롯 매핑 (top/mid/content/bottom/side-left/right…) 을 객체 기반 설정 맵으로 단일화. fluid layout key 를 컨테이너 폭에 따라 auto/fluid 로 동적 전환 (adManager.js:48-72), shimmer placeholder + sessionStorage 5분 캐시로 LCP 보호.
  • 전략적 가치: 저ARPU 트래픽(동남아·국내)은 AdSense 로 수익화, 고LTV 사용자는 Xsolla 젬 결제로 전환시키는 이중 funnel 을 단일 사용자 세션에서 동시 운영 가능한 구조. CFR-protected 콘텐츠는 consumeGemForContent callable 로 게이팅하여 광고 회피 사용자에게서도 매출이 발생하도록 설계.

2. 기술적 고도화 및 최적화 성과

스택 (실측 기반)

  • Vite 7.3 + Firebase v12 (Modular API) + Firebase Functions v2 (2nd gen) + Cloudflare Workers + Phaser 3 + Spine v3.

빌드/번들 최적화

  • Vite rollupOptions.input 에 16개 HTML 엔트리 등록 (MPA 구조).
  • manualChunks 로 Firebase 를 firebase (app/firestore/auth/storage) 와 firebase-extra (analytics/database) 로 분리 — Auth 가 필요 없는 board/download 페이지에서 analytics 청크 다운로드를 회피.
  • viteStaticCopy 로 games/ 폴더(Phaser 빌드)를 별도 번들 라이프사이클로 격리하여, 메인 SPA 빌드 변경이 게임 자산에 영향 주지 않도록 함.

SEO/SSR 하이브리드 functions/index.js viewSSR

  • 콘텐츠 상세 페이지 (view.html) 에 대해 Cloud Function 기반 SSR 렌더러 구현: Firestore 5초 타임아웃 + 메모리 LRU (200건/TTL 5분) + 디스크 스냅샷 (33개) 3중 캐시 fallback. CDN s-maxage=3600 을 헤더에 설정해 Cloud Function 호출 단가를 최소화 (functions/index.js:24-79, 304-305 주석).
  • 사전 렌더링: npm run build:snapshots 빌드 단계에서 33개 콘텐츠 스냅샷을 view-snapshots/ 로 정적 출력 → AdSense 심사·구글봇 호환을 SPA 의 단점 없이 확보.

아키텍처 계층화

  • src/ (4,973 LOC) — UI/유틸/Firebase 어댑터 (firebase/{auth, config, rtdb}.js)
  • managers/ (1,179 LOC) — 도메인 매니저 (Ad/Badge/Xsolla) 로 횡단 관심사 격리
  • functions/ (1,403 LOC) — 10개 Cloud Function (결제/SSR/관리자/배치/감사로그)
  • data/ — 콘텐츠/번역/뱃지 메타데이터 (SSOT)

5차에 걸친 점진적 리팩토링 (메모리 기록 기반, 18일 전 시점 검증 필요)

  • adManager.js 478→300 LOC (-37%, if-else→객체맵, 헬퍼 5개 추출)
  • doll.css 392→0 LOC (override 분리 후 완전 제거)
  • auth.js initAuth() 78 LOC → 3개 단일 책임 함수로 분해
  • 메모리 누수 5건, 인라인 컬러→CSS Custom Property 6종, transition: all 34곳, font-family 144개 중복 제거
  • pushIfReady() Firebase RTDB 가드 래퍼 도입으로 레이스 컨디션 차단

3. 글로벌 서비스 운영 및 비용 효율화

Cloudflare Workers 에셋 게이트웨이 cloudflareWorker.js

  • Hot-link / 무단 다운로드 차단: Referer 화이트리스트 (armii-treasure.com, armii-paper.com, armiicraft.com …) 통과 시에만 오리진 히트.
  • Edge Rate Limiting: IP당 분당 2회 (Map 기반, 10,000 엔트리 초과 시 자동 GC) — DDOS·크롤러 비용 누수를 Cloud Function 진입 전 차단.
  • Path Obfuscation: base64 인코딩된 ?q= 파라미터로 오리진 경로 마스킹 → 정적 자산 URL 추측 공격 무력화.
  • Origin Header Hardening: X-Powered-By/Server 제거, Content-Disposition 동적 주입으로 다운로드 UX 와 보안 동시 확보.

서버리스 비용 최적화 4종

  1. viewSSR 3중 캐시 (CDN edge 1h → Function 메모리 LRU 5분 → Firestore 5s timeout → 정적 스냅샷) → cold path 만 Firestore read 발생 (functions/index.js:304-305).
  2. flushAnalyticsBuffer + scheduledFlush: 클라이언트 분석 이벤트를 RTDB 버퍼에 누적 후 스케줄 함수로 일괄 Firestore commit → 쓰기 비용을 N→1 로 압축.
  3. Firestore manual chunk + lazy firebase/database: 페이지별로 SDK 모듈 동적 로딩.
  4. Region 단일화 (asia-southeast1) — 한국·일본·동남아 타겟에 최저 RTT 인 싱가포르 리전으로 확정.

보안·운영 거버넌스

  • Firestore Rules (8.1 KB, phase1 백업 보존), admins/{uid} + role-claim 기반 RBAC, assertAdmin() 헬퍼로 중요 함수 7개 진입점 통합 검증, adminWriteAuditLog 로 감사 추적.
  • ko/en/ja 3개 로캘 i18n + getLang() 단일 진입점 통합으로 다국어 분기 누수 차단.

4. AI 기반 생산성 및 자동화 역량

검증된 사실

  • Claude Code 기반의 영구 메모리 시스템 (.claude/projects/.../memory/) 을 도입하여 5차에 걸친 리팩토링 이력 (완료 일자·LOC 변화·보류 항목·Why/How to apply 메타데이터 포함) 을 자가 누적. 후임 인계·세션 간 컨텍스트 손실 방지.
  • AdSense 심사 정책과 충돌하는 인라인 스크립트 분리 작업을 "심사 통과 전까지 보류" 라는 정책 메모리로 명시적으로 게이트 — AI 가 작업 우선순위 판단까지 위임받는 구조.
  • 16개 HTML 엔트리·1,400 LOC Cloud Functions·600 LOC PG 통합 등 MPA + 서버리스 + PG 통합의 풀스택 범위를 1인이 운영 — AI 페어 프로그래밍 없이 통상 다인 팀 규모.
주의 — 정직 모드
  • "리드타임 N% 단축" 같은 수치는 측정 인프라가 없어 임의 수치 기재는 위험. 면접에서 백업 자료 요구 받으면 무너짐. "AI 페어 프로그래밍 워크플로우를 통해 5차 리팩토링 사이클을 단독으로 완수" 정도로 사실 기반 톤다운 권장.
  • Bedrock·Gemini CLI 는 사용 흔적이 없으니 이력서에서 제외, 만약 실제로 사용했다면 그 결과물 (생성된 자산/스크립트) 을 레포에 커밋해 증거로 남기는 게 우선.
