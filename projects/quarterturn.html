QUARTER TURN — 기술 상세

호스트-게임 분리 아키텍처, dia 통합 화폐 SDK, 레거시 리마스터 파이프라인 노트

2026.05 — 현재 · quarterturn.pages.dev ↗
한국어 전용 · Korean only
육아휴직 기간에 시작한 1인 프로젝트. 브라우저에서 바로 실행되는 HTML5 게임 포털이지만, 실제로 만든 것은 게임 한 편이 아니라 게임을 계속 얹을 수 있는 파이프라인이다. 호스트가 지갑·세이브·랭킹을 소유하고 게임은 iframe 으로 붙는 구조를 잡아, 현재 14종이 같은 규약 위에서 돌아간다.

1. 호스트-게임 분리 아키텍처

왜 분리했나

  • 게임마다 저장 방식·화폐·랭킹을 따로 만들면 N개의 게임은 N개의 백엔드 결정이 된다. 대신 호스트가 계정·지갑·세이브·랭킹을 전부 소유하고, 게임은 정해진 API 만 호출하는 손님으로 두었다.
  • 게임은 iframe 으로 마운트되고 호스트와는 postMessage 로만 대화한다 (src/lib/walletBridge.js). 요청마다 reqId 를 붙인 비동기 왕복 구조라, 나중에 백엔드가 동기 → 비동기로 바뀌어도 게임 쪽 코드는 그대로다.
  • iframe → 호스트 왕복 실측 약 16ms. 결제 팝업(autoShop)이 브라우저의 user-gesture 컨텍스트를 잃지 않는지까지 popup blocker 환경에서 확인했다.

Pakedev SDK public/vendor/pakedev-sdk.js (818 LOC)

  • 게임 작성자가 보는 유일한 표면. Pakedev.wallet(잔액 조회/차감/변경 구독), Pakedev.storage(키 단위 세이브), Pakedev.ranking(점수 제출), Pakedev.shop(충전 진입) 네 갈래로 정리했다.
  • 차감만 게임에게, 충전은 호스트에게. 화폐 발행 경로를 호스트 한 곳으로 묶어두면 나중에 서버 권위로 전환할 때 손댈 지점이 하나로 수렴한다.
  • SDK 가 없는 환경(게임 단독 실행)에서도 게임이 죽지 않도록 dev 폴백을 규약에 포함. 게임 개발자는 호스트 없이 자기 게임만 띄워놓고 작업할 수 있다.

2. dia 통합 화폐와 백엔드 교체

단계적 전환 설계

  • Phase 1 — 클라이언트 localStorage 권위. 지갑·거래내역·HUD·충전 페이지까지 먼저 완성해 UX 와 API 계약을 확정했다.
  • Phase 3 — Firebase Anonymous Auth + Firestore(asia-northeast3) 로 교체. wallets/{uid} 는 onSnapshot 라이브 구독, 거래는 transactions/{uid}/items/{txId} 에 txId 를 문서 ID 로 써서 멱등 보장, 세이브는 saves/{uid}/games/{gameId} 게임당 1문서에 키 단위 merge / deleteField.
  • 교체 결과 게임/SDK 코드 변경 0건. wallet.js 는 공개 API 의 동기 시그니처를 그대로 유지한 채 내부에만 Firestore 미러를 붙였고, 브리지가 이미 비동기라 게임 쪽은 차이를 느끼지 못한다. HUD·상점·브리지 모두 무수정 통과.
  • 모든 store 는 DB 미설정 시 localStorage 폴백을 유지한다. 설정 하나로 오프라인/로컬 모드가 되고, prod 전환은 Cloudflare Pages 환경변수(VITE_FIREBASE_*) 스왑만으로 이뤄진다 — 코드 분기 없음.
  • Firestore Security Rules 를 저장소에 수기로 관리하고 (firebase deploy --only firestore:rules 타깃 지정 배포), 양성(본인 지갑 읽기 허용) · 음성(타인 지갑 읽기 차단) 스모크 테스트로 규칙이 실제로 강제되는지 확인했다.

3. 레거시 리마스터 파이프라인

문제

되살리고 싶은 옛 프로젝트는 대부분 소스와 에셋만 남고 빌드가 되지 않는 상태였다. 엔진 버전, 유료 플러그인, 사라진 SDK — 원본을 다시 빌드하는 길은 막혀 있었다.

해법 — 절차를 문서로 고정 UNITY-REMASTER-PLAYBOOK.md

  • 원작이 진실원이다. 스탯·가격·확률·UI 배치·스킬 동작을 추측으로 재구현하면 거의 다 틀린다는 걸 첫 프로젝트에서 확인하고, "막히면 legacy 를 grep/덤프부터" 를 첫 번째 규칙으로 박았다. (보스 등장 조건이 "입장 즉시" 가 아니라 "잡몹 처치 게이지 충족 시" 였던 식의 실제 오답 사례를 함께 기록.)
  • 추출은 전부 스크립트로. Unity 의 *.prefab(UI 앵커/좌표/스프라이트), Assets/Script/**/*.cs(전투식·비용 공식·스폰), 로컬라이제이션 테이블, 그리고 BGDatabase 바이너리(*.bytes) 까지 파서를 직접 짜서 src/data/extracted/*.json 을 만든다. legacy 를 다시 받아도 재실행하면 동일 결과가 나온다.
  • "export 해줄 사람이 없어서 못 살린다" 고 미뤄뒀던 밸런스 수치를 바이너리에서 자체 추출해 복구한 게 이 파이프라인의 첫 성과였다.
  • 이식 타깃은 TypeScript + PixiJS v8 + DOM UI. 렌더링은 Pixi, UI 는 DOM 으로 갈라 레이아웃 작업을 웹 표준 쪽으로 넘겼다.
  • 검증 가능한 단위로 쪼개고 청크마다 커밋. "폭발/투척 FX", "슬래시 FX" 처럼 한 커밋 = 한 기능 + 그 검증. 실제 게임 화면으로 눈으로 확인한 것만 완료로 친다.

이 플레이북 덕분에 두 번째 이후 프로젝트는 같은 함정을 다시 밟지 않는다. 현재 리마스터·신작 9종이 이 절차 위에서 진행 중이다.

4. 랭킹 무결성 — 신뢰 등급을 먼저 정했다

  • 랭킹은 붙이기는 쉽지만 클라이언트 권위인 채로 "랭킹이 있다" 고 말하면 거짓말이 된다. 그래서 게임별 치트 표면을 먼저 감사해 보드를 검증가능성 클래스로 분류했다 (docs/INTEGRITY_AUDIT.md → docs/RANKING_AND_INTEGRITY.md).
  • progression 보드 우선(도달 스테이지 등). 점수성 보드보다 서버 검증이 깨끗하고, 나중에 서버 권위로 전환할 때 재작업 부담이 없다.
  • 서버 검증 전까지 점수성·P2W 보드는 "켜되 신뢰하지 않음" 으로 명시. 켜는 이유는 지표와 UI 확정이고, 그 상태로 순위의 정당성을 주장하지 않는다.
  • 세이브 변조 위협 모델도 같은 방식으로 문서화 — 호스트 단일 지점 서명, 영수증 파생 보유(구매 아이템은 거래 내역에서 재구성), 비정상 점프 감사 추적. 실행은 서버 권위 전환과 묶여 있고, 그 전까지는 결정과 근거만 남겨두었다.

5. 운영을 코드와 문서로 강제하기

  • 런칭 게이트(docs/LAUNCH_CHECKLIST.md) — 개발 편의 장치(초기 dia 자동 지급, 무료 충전, 전역 디버그 객체)가 실서비스로 새어나가지 않도록 파일:라인 단위로 목록화. 항목마다 "왜 지금은 못 지우는지" 의 선후관계까지 적었다.
  • 권리 정리 게이트 — 권리 확인이 끝나지 않은 타이틀은 코드네임으로 격리하고 locked: true 소프트 게이트로 공개 목록에서 차단한다. 게임 내부 문자열·탭 제목까지 구명 노출을 제거했다. 정책을 사람 기억이 아니라 데이터 필드로 만들어 배포 파이프라인이 강제하게 한 부분.
  • 규제 선행 검토 — 가상화폐(dia) 의 선불수단 포섭 여부·환불 강행규정·등급분류·PG 이슈를 실결제 연동 전에 상담 체크리스트로 정리(docs/LEGAL_CONSULT_CHECKLIST.md).
  • AI 운영 보조의 권한 경계 설계(docs/AI_STAFF.md) — LLM 은 복잡도 임계를 넘으면 붕괴하고 단일 환각이 멀티스텝을 깨뜨린다는 전제(95% 정확도 × 20스텝 = 36%)에서 출발해, 읽기·분류·요약·초안은 자동, 공개 포스팅·제재·결제·외부 발송은 전부 사람 게이트로 선을 그었다. 모든 공개용 산출물에는 [DRAFT — 사람 검토 대기] 머리표를 강제한다.

6. 그 외 구현 메모

  • Vite MPA(vanilla JS) — 홈 / 게임뷰 / 상점 3 엔트리. 호스트 코드 약 3,700 LOC.
  • Phaser 3 를 public/vendor/ 공유본으로 두고 iframe 마다 HTTP 캐시로 재사용 — 게임별 중복 번들 제거.
  • 게임이 자체 FIT 스케일을 구현하지 않은 경우에만 호스트가 baseViewport 로 고정 크기 + transform scale 축소. 게임이 FIT 을 구현해 재납품하면 이 예외를 제거하는 방향.
  • 모바일에서 게임 화면 스케일이 출렁이거나 가상 자판이 뷰포트를 줄이는 문제를 게임뷰 레이어에서 차단.
  • ko/en 런타임 i18n, PWA manifest, 신규 게임 스캐폴딩 CLI(npm run new-game)와 등록 검증 스크립트(validate-games).
주의 — 정직 모드
  • 서비스 중이 아니라 공개 개발 중이다. 등록 14종 중 완성 5종, 나머지 9종은 개발 중이며 그중 6종은 비밀번호 게이트로 잠겨 있다. 런칭 체크리스트도 아직 미완이라 개발 편의 장치가 코드에 남아 있다.
  • DAU·매출·리텐션 같은 지표는 없다. 실결제 연동 전이고 마케팅도 시작하지 않았다. 이 프로젝트에서 볼 것은 트래픽이 아니라 아키텍처 결정과 그 근거다.
  • 랭킹은 서버 검증 전이라 순위의 신뢰성을 주장하지 않는다. 위 4장의 분류가 그 이유다.
