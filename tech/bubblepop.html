📋 프로젝트 개요

Bellygom Bubble Pop은 Phaser 3 기반의 크로스 플랫폼 버블 슈팅 퍼즐 게임으로, Android, Toss 미니게임, webOS TV를 지원하며 특히 저사양 LED TV 환경 최적화에 집중한 프로젝트입니다.

출시 이력

  • 2025.12 - Android 출시 (Google Play)
  • 2025.12 - Toss Mini 출시
  • 2026.02 - webOS TV 출시 (LG Content Store)

기술 스택

  • 엔진: Phaser 3.90
  • 언어: TypeScript
  • 빌드: Vite + Legacy Plugin
  • 백엔드: Firebase (Authentication, Analytics, Firestore)
  • 플랫폼: webOS TV (24/25/26), Toss Mini, Android, Mobile Web, Desktop

핵심 도전 과제

  • LED TV 저사양 환경 (RAM 1.5GB) 안정화
  • OLED/LED TV 호환성 확보
  • LG 자동화 심사 환경(Tellurium) 대응
  • 메모리 사용량 200MB → 60MB 최적화

1. Manager 패턴 (Singleton)

각 도메인별 단일 책임 매니저로 전역 상태를 관리합니다.

// 각 도메인별 단일 책임 매니저
export class DataManager {
  private static instance: DataManager;
  
  static getInstance() {
    if (!this.instance) {
      this.instance = new DataManager();
    }
    return this.instance;
  }
  
  // 플레이어 데이터 관리
  // 로컬 스토리지 동기화
  // 클라우드 백업 (Firebase)
}

// 기타 매니저들
- SoundManager: 사운드 풀링 및 재생
- AdManager: 광고 통합 (AdMob)
- PlatformManager: 플랫폼별 분기 처리
- PaymentManager: 인앱 결제 통합
장점:
• 전역 상태 관리 일관성
• 씬 간 데이터 공유 용이
• 메모리 효율적 (인스턴스 1개)

2. Scene 구조 (FSM 패턴)

BootScene (초기화)
    ↓
LoadingScene (에셋 로딩)
    ↓
TitleScene (메인 메뉴)
    ↓ ← → ShopScene, OptionScene
GameScene (실제 게임)
    ↓
ResultScene (결과)

특징:

  • 각 씬은 독립적으로 동작
  • 씬 전환 시 메모리 자동 해제
  • 모달 씬은 launch/stop 사용

3. 메모리 최적화 전략

A. Lazy Loading

// BootScene: 필수 에셋만
this.load.image('logo', 'logo.png');

// TitleScene 진입 시
this.load.atlas('ui', 'ui.png', 'ui.json');

// GameScene 진입 시
this.load.atlas('game', 'game.png', 'game.json');

B. 텍스처 압축

단계 크기 감소율
원본 PNG (2048x2048) 4MB -
TinyPNG 압축 1.2MB -70%
WebP 변환 400KB -90%
메모리 (압축 해제) 16MB -

C. 에셋 분할 로딩

// TV: 현재 챕터만
if (platformManager.isTV) {
  loadChapter(currentChapter);
}
// Mobile: 현재 + 다음 챕터
else {
  loadChapter(currentChapter, currentChapter + 1);
}

4. 플랫폼별 분기 처리

export class PlatformManager {
  isTV = import.meta.env.VITE_TARGET_ENV === 'tv';
  isToss = import.meta.env.VITE_TARGET_ENV === 'toss';
  isAndroid = import.meta.env.VITE_TARGET_ENV === 'android';
  isMobile = /Mobile|Android|iPhone/.test(navigator.userAgent);
  
  // webOS API 감지
  hasWebOS = typeof window.webOS !== 'undefined';
  
  // 메모리 프로파일
  getMemoryProfile() {
    if (this.isTV) {
      return {
        textureLimit: 512,  // 512x512 max
        audioChannels: 8,
        particleLimit: 50
      };
    }
    return {
      textureLimit: 2048,
      audioChannels: 32,
      particleLimit: 200
    };
  }
}

🐛 트러블슈팅 사례

문제 1: webOS 24 LED TV 크래시

증상:
• OLED TV: 정상 작동
• LED TV (43UT9000PUA): 로딩 중 멈춤

원인 분석:

LED TV (43UT9000PUA):
- RAM: 1.5GB
- 사용 가능: ~300MB
- 게임 메모리: 200MB
→ OOM (Out of Memory)

해결:

  1. 이미지 압축 (TinyPNG → 70% 감소)
  2. Lazy loading (BootScene 에셋 최소화)
  3. 초기 대기 시간 증가 (3초 메모리 안정화)
  4. 청크 분할 (Phaser 엔진 별도 파일)
✅ 결과: LED TV 안정 작동 (13번 리젝 후 승인)

문제 2: Tellurium 자동화 환경

증상:
• 수동 테스트: 정상
• LG 자동화 심사: 실패

원인:

// Tellurium이 window.load 이벤트 가로챔
window.addEventListener('load', initGame); // ❌ 실행 안 됨

해결:

// DOMContentLoaded + Failsafe
if (document.readyState === 'loading') {
  document.addEventListener('DOMContentLoaded', init);
} else {
  setTimeout(init, 500);
}

// 5초 후 강제 실행
setTimeout(() => {
  if (!initialized) init();
}, 5000);
✅ 결과: Tellurium 환경에서 안정적으로 시작

💡 핵심 인사이트

  1. 저사양 최적화는 고사양도 개선
    LED TV를 위한 최적화가 모든 기기의 성능을 향상시킴
  2. 에셋 최적화 > 코드 최적화
    코드 최적화: 20% 개선 / 에셋 최적화: 70% 개선
  3. Graceful Degradation
    기능 감소보다 안정성 우선. TV에서는 낮은 품질이라도 크래시보다 낫다.
  4. 플랫폼별 타겟팅
    단일 코드베이스로 5개 플랫폼 지원. 빌드 환경 변수로 분기 처리.
